There are a few ways these meetings might be organized depending on where we are in the design-development process.<\/p>\n\n\n\n

  • early concept<\/strong>: help me think about the problem<\/li>
  • late concept<\/strong>: help me choose among these ideas and develop one<\/li>
  • early detail<\/strong>: this is where we are going, what caveats remain?<\/li>
  • late detail<\/strong>: here’s a the detail, what questions do I need to address?<\/li><\/ul>\n\n\n\n

    These depend on the thought that there are a few phases\/horizons to product design work, namely <\/p>\n\n\n\n

    1. Problem Framing (aka Research or Horizon 3), in which we learn about the challenges and needs of users to understand and select a problem to go after with concepts,<\/li>
    2. Concept (or Horizon 2), in which we develop concepts and select among them with the help of users and engineering, and<\/li>
    3. Detail (or Horizon 1), in which we develop the selected concept and test the fitness of our more detailed work with users as we go.<\/li><\/ol>\n\n\n\n

      It’s a common anti-pattern that engineering gets involved only when we are in detail, but we’d like to pull them into the other phases to be informed about the available problems and to help us develop and select among concepts.<\/p>\n\n\n\n

      Some Agile purists will decry this phasing as anti-agile, but that requires a blinkered view of what Agile means, which often comes about when one confuses Scrum for Agile.
